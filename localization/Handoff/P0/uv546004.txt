	<title>Power BI Analysis Services Connector in-depth</title>
	
	<div><font face="Times New Roman" size="3">

</font><p style="margin: 6pt 0in;"><font face="Calibri" size="3">With Power BI, users in your organization can now
see your on-premises SQL Server Analysis Services data and metrics in a whole
new way. But before users can connect to your Analysis Services models as a
data source, an Analysis Services Connector needs to be installed and
configured. The connector helps make the behind-the-scenes communication from a
user in Power BI to your on-premises Analysis Services server and back to Power
BI quick and secure. </font></p><font face="Times New Roman" size="3">

</font><p style="margin: 6pt 0in;"><font face="Calibri" size="3">Installing and configuring a connector is usually done by an
administrator. It requires special knowledge of your Analysis Services servers
and it requires Server Administrator permissions. </font></p><font face="Times New Roman" size="3">

</font><p style="margin: 6pt 0in;"><font face="Calibri" size="3">This article doesn’t provide step-by-step guidance on how to
install and configure an Analysis Services Connector. For that, be sure to see </font><a href="http://support.powerbi.com/knowledgebase/articles/471577-configure-a-power-bi-analysis-services-connector-f"><font color="#0000ff" face="Calibri" size="3">Configure
a Power BI Analysis Services Connector</font></a><font face="Calibri" size="3">. This article is meant to provide
you with an in-depth understanding of how the connector works. We’ll also go
into some detail about usernames and security in both Power BI and Analysis
Services, and how Power BI uses the e-mail address a user signs into Power BI
with, the connector, and Active Directory to securely connect to and query your
on-premises Analysis Services data. If you want to dig deeper, links to more
detailed articles are provided.</font></p><p style="margin: 6pt 0in;"><font face="Times New Roman" size="3"><br></font></p><h1 style="margin: 12pt 0in 6pt;"><font color="#365f91" face="Cambria" size="5">Making the connection</font></h1><div style="margin: 6pt 0in;"><font face="Times New Roman" size="3">

</font><font face="Calibri" size="3">Let’s first look at what happens when a user in Power BI interacts with a report connected to an Analysis Services datasource. </font></div><div style="margin: 6pt 0in;"><font face="Times New Roman" size="3"></font><font face="Calibri" size="3">1.&nbsp; When a user interacts with a report in Power BI, such as click on a chart, select a filter, or add a new field, a Data Analysis Expressions (DAX) query is created by Power BI and sent along with an effective username to the Analysis Services Connector.</font></div><div style="margin: 6pt 0in;"><font face="Times New Roman" size="3"></font><font face="Calibri" size="3">2.&nbsp; The Analysis Services Connector acts as a bridge. It sends the DAX query and effective username along to the Analysis Services server. </font></div><div style="margin: 6pt 0in;"><font face="Times New Roman" size="3"></font><font face="Calibri" size="3">3.&nbsp; Because the Analysis Services server is domain joined, it sends the effective username to an Active Directory server in that same domain.</font></div><div style="margin: 6pt 0in;"><font face="Times New Roman" size="3"></font><font face="Calibri" size="3">4.&nbsp; The Active Directory server returns a qualified Windows username (domain\username) to the Analysis Services server.</font></div><div style="margin: 6pt 0in;"><font face="Times New Roman" size="3"></font><font face="Calibri" size="3">5.&nbsp; The Analysis Services server checks to see if the username belongs to a Role with at least Read permissions for the Tabular model. If they do,&nbsp;the DAX query is run.</font></div><div style="margin: 6pt 0in;"><font face="Times New Roman" size="3"></font><font face="Calibri" size="3">6.&nbsp; The result from the DAX query is returned from Analysis Services, to the Analysis Services Connector, and then onto Power BI where it’s rendered in the user’s report.</font><font face="Times New Roman" size="3"><br></font></div><font face="Calibri" size="3"><p style="margin: 6pt 0in;">Now, let’s go over what needs to be in-place for this to
happen.<br></p><font face="Times New Roman"><h1>

<font color="#365f91" face="Cambria" size="5">Usernames in Power BI</font></h1><p style="margin: 6pt 0in;">

<font face="Calibri">When a user signs up for a Power BI account, they use an e-mail
address. In most cases, this is a work e-mail address, like nancy@contoso.com. </font></p><p style="margin: 6pt 0in;">

<font face="Calibri">However, in some cases, it could be an e-mail address like nancyt@contoso.onmicrosoft.com.
This is most common when a user signs up for Office365 and doesn’t use their
work e-mail address. This is an important distinction and we’ll come back to it
later when we talk about when you might need to synchronize your Active
Directory with Azure Active Directory by using DirSync.</font></p><p style="margin: 6pt 0in;">

<font face="Calibri">The e-mail address users sign into Power BI with is known as
an <i style="mso-bidi-font-style: normal;">effective username</i>. In-fact, when
connecting to Analysis Services, it’s a value in an EffectiveUserName
connection string property. Each time a user interacts with a report connected
to Analysis Services, the effective username is passed to the
Analysis Services Connector and then onto your on-premises Analysis Services
server. It must then be converted by an Active Directory server in the same
domain to a Windows domain\username account specified for the roles that Analysis
Services use to secure your data. </font></p><p style="margin: 6pt 0in;"><br></p><h1 style="margin: 12pt 0in 6pt;"><font color="#365f91" face="Cambria" size="5">Usernames in Analysis Services</font></h1><p style="margin: 6pt 0in;">

<font face="Calibri">Before we talk about how an effective username and the connector
is used to connect to Analysis Services, let’s take a quick look
at how Analysis Services secures data. In tabular models, there are two levels
of security:</font></p><p style="margin: 6pt 0in;">

<font color="#365f91" face="Cambria">Role-based security</font></p><p style="margin: 6pt 0in;">

<font face="Calibri">Tabular models provide security based on user roles. Roles
are defined for a particular tabular model project during authoring in SQL
Server Data Tools – Business Intelligence (SSDT-BI), or after a model is
deployed, by using SQL Server Management Studio (SSMS). Roles contain members
by Windows username or by Windows group. Roles define permissions a user has to
query or perform actions on the tabular model. Most users will belong to a role
with Read permissions. Other roles are meant for administrators with
permissions to process tables, manage database functions, and manage other
roles. </font></p><p style="margin: 6pt 0in;">

<font color="#365f91" face="Cambria">Row-level security</font></p><p style="margin: 6pt 0in;">

<font face="Calibri">Tabular models also provide dynamic, row-level security.
Unlike having at least one role in which users belong to, dynamic security is
not required for any tabular model. At a high-level, dynamic security defines a
users read access to data right down to a particular row in a particular table.
Similar to roles, dynamic row-level security relies on a user’s Windows
username.</font></p><p style="margin: 6pt 0in;">

<font face="Calibri">In Power BI, a users ability to query and view tabular model
data are determined first by the roles their Windows user account are a member
of and second, by dynamic row-level security, if configured.</font></p><p style="margin: 6pt 0in;">

<font face="Calibri">Implementing role and dynamic row-level security in tabular
models is beyond the scope of this article.<span style="mso-spacerun: yes;">&nbsp;
</span>You can learn more at </font><a href="https://msdn.microsoft.com/en-us/library/hh213165.aspx"><font color="#0000ff" face="Calibri">Roles (SSAS
Tabular)</font></a><font face="Calibri"> on MSDN. And, for the most in-depth understanding of tabular model
security, download and read the </font><a href="https://msdn.microsoft.com/en-us/library/jj127437.aspx"><font color="#0000ff" face="Calibri">Securing the
Tabular BI Semantic Model</font></a><font face="Calibri"> whitepaper. </font></p><p style="margin: 6pt 0in;"><strong>

</strong>

</p><h1 style="margin: 12pt 0in 6pt;"><font color="#365f91" face="Cambria" size="5">What is Active Directory’s role?</font></h1><p style="margin: 6pt 0in;">

</p><p style="margin: 6pt 0in;"><font face="Calibri">In order for Analysis Services Server to determine if a user
connecting to it belongs to a role with permissions to read data, the server
needs to convert the effective username passed from Power BI to the connector
and then onto the Analysis Services server. The Analysis Services server passes
the effective username to a Windows Active Directory server <i style="mso-bidi-font-style: normal;">joined to the same domain</i>. The Active
Directory server then validates the effective username and returns that user’s
Windows username back to the Analysis Services server. </font></p><p style="margin: 6pt 0in;"><strong>

</strong>

</p><h1 style="margin: 12pt 0in 6pt;"><font color="#365f91" face="Cambria" size="5">What about Azure Active Directory?</font></h1><p style="margin: 6pt 0in;">

</p><p style="margin: 6pt 0in;"><font face="Calibri">Because Power BI is a cloud service, it uses </font><a href="http://azure.microsoft.com/en-us/documentation/articles/active-directory-whatis/"><font color="#0000ff" face="Calibri">Azure
Active Directory</font></a><font face="Calibri"> to take care of authenticating users. </font></p><p style="margin: 6pt 0in;">

</p><p style="margin: 6pt 0in;"><font face="Calibri">When users connect to an on-premises Analysis Services
server from Power BI, their username must also be resolved in your domain’s
Active Directory. If users in your organization login with their work e-mail
address, like nancy@contoso.com, username resolution typically isn’t a problem.
In some cases, if a user logs into Power BI with an .onmicrosoft.com address,
your domains Active Directory server might need to be synchronized with Azure
Active Directory. Fortunately, there’s a tool just for this.</font></p><p style="margin: 6pt 0in;"><strong>

</strong>

</p><h1 style="margin: 12pt 0in 6pt;"><font color="#365f91" face="Cambria" size="5">Synchronize an on-premises Active Directory with Azure Active Directory</font></h1><p style="margin: 6pt 0in;">

</p><p style="margin: 6pt 0in;"><font face="Calibri">Synchronizing your Active Directory to Azure Active
Directory with the Directory Synchronization (DirSync) tool is not a
requirement if users sign into sign into Power BI with their work e-mail
address. But&nbsp;if users sign into Power BI with a .onmicrosoft.com email
address, you will need to sync your Active Directory server with Azure Active
Directory.</font></p><p style="margin: 6pt 0in;">

</p><p style="margin: 6pt 0in;"><font face="Calibri">Let's assume your Analysis Services server and Active
Directory server are joined to the same domain, Contoso.com, and an Analysis
Services Connector is installed, configured and running. What happens when a
user, Nancy, signs into Power BI with...</font></p><p style="margin: 6pt 0in;">

</p><p style="margin: 6pt 0in;"><b style="mso-bidi-font-weight: normal;"><font face="Calibri">A work email address
like nancy@contoso.com</font></b></p><p style="margin: 6pt 0in;">

</p><p style="margin: 6pt 0in;"><i style="mso-bidi-font-style: normal;"><font face="Calibri">You do not
need&nbsp;to sync your Active Directory with Azure Active Directory. </font></i></p><p style="margin: 6pt 0in;">

</p><p style="margin: 6pt 0in;"><font face="Calibri">When Nancy interacts with a report connected to an
Analysis Services datasource, the value of the effective username passed with
the DAX query is 'nancy@contoso.com'. Active Directory is able to resolve Nancy’s
username and the DAX query is run in Analysis Services. Nancy is able to
interact with her reports connected to Analysis Services without any problems. No
Active Directory to Azure Active Directory sync is needed. However, keep the
following in-mind:</font></p><ul><li><div style="margin: 6pt 0in;"><font face="Calibri">Your&nbsp;on-premises Analysis Services&nbsp;server must be joined to a domain, and your Active Directory server must also be joined to that same domain. </font></div></li><li><div style="margin: 6pt 0in;"><font face="Calibri">If your Analysis Services server is part of a workgroup, Active Directory doesn’t come into the picture at all. The effective username cannot be resolved and users will be unable to connect to the server.</font></div></li></ul><p style="margin: 6pt 0in;"><b style="mso-bidi-font-weight: normal;"><font face="Calibri">A .onmicrosoft.com
email address, like nancyt@contoso.onmicrosoft.com</font></b></p><p style="margin: 6pt 0in;">

</p><p style="margin: 6pt 0in;"><i style="mso-bidi-font-style: normal;"><font face="Calibri">You will need to sync
your on-premises Active Directory with Azure Active Directory. </font></i></p><p style="margin: 6pt 0in;">

</p><p style="margin: 6pt 0in;"><font face="Calibri">Nancy’s work e-mail is nancy@contoso.com, but when she signs
into Power BI, she uses a different e-mail address,
nancyt@contoso.onmicrosoft.com. When Nancy interacts with a report connected to
an on-premises Analysis Services tabular model, the effective username passed
with the DAX query is nancyt@contoso.onmicrosoft.com. The on-premises Active
Directory won’t be able to resolve this username to Nancy’s on-premises
username. Analysis Services cannot identify Nancy’s membership in a role and
the query will not be run. </font></p><p style="margin: 6pt 0in;">

</p><p style="margin: 6pt 0in;"><font face="Calibri">In this case, you can use DirSync to synchronize Active Directory
to Azure Active Directory. Once synchronized, Nancy can sign into Power BI
using her work e-mail address, nancy@contoso.com. Nancy will be able to
interact with her reports just like in our scenario above.</font></p><p style="margin: 6pt 0in;">

</p><p style="margin: 6pt 0in;"><font face="Calibri">To learn more about using DirSync, see </font><a href="https://technet.microsoft.com/en-us/library/jj573653.aspx"><font color="#0000ff" face="Calibri">Directory
Integration</font></a><font face="Calibri"> on TechNet.</font></p><p style="margin: 6pt 0in;"><strong>

</strong>

</p><h1 style="margin: 12pt 0in 6pt;"><font color="#365f91" face="Cambria" size="5">Now, this is where the Analysis Services Connector comes in</font></h1><p style="margin: 6pt 0in;">

</p><p style="margin: 6pt 0in;"><font face="Calibri">The connector acts as a bridge between the Power BI cloud
service running in a user’s browser and your on-premises Analysis Services
server. Data transfer between Power BI and the Analysis Services Connector is
secured through </font><a href="http://azure.microsoft.com/en-us/documentation/services/service-bus/" target="_blank"><font color="#0000ff" face="Calibri">Azure Service Bus</font></a><font face="Calibri">. The Service Bus creates a secure channel
between<span style="mso-spacerun: yes;">&nbsp; </span>Power BI and your on-premises Analysis
Services server through an outbound connection.&nbsp; An inbound connection is
not required to be opened in your on-premises firewall. </font></p><p style="margin: 6pt 0in;">

</p><p style="margin: 6pt 0in;"><font face="Calibri">You’ll need to install the connector on a computer joined to
the same domain as your Analysis Services server and Active Directory. This can
be the same computer as Analysis Services. In-fact, it’s even better because
the communications between the connector and your Analysis Services server doesn’t
have to make an extra hop over the network.</font></p><p style="margin: 6pt 0in;"><strong>

<br></strong>

</p><h2 style="margin: 2pt 0in 0pt;"><font color="#365f91" face="Cambria" size="4">Before you install and configure a connector, be sure you have the
following:</font></h2><ul><li><div style="margin: 6pt 0in;">

<font face="Calibri">The name of the Analysis Server you want to connect to. You will specify it right away when you step through the Connector’s configuration wizard. If your server uses a default instance name, you’ll only need to enter that name. If you’re connecting to an instance, you’ll enter ServerName\Instance.</font></div></li><li><div style="margin: 6pt 0in;"><font face="Calibri">A Windows username and password with Server Administrator permissions for the instance you’re connecting to. Be sure this account has a password that doesn’t expire. If it does, users will get a connection error.</font></div></li><li><div style="margin: 6pt 0in;"><font face="Calibri">Users will need to belong to a Analysis Services Role with at least Read permissions configured for the models they are connecting to. </font></div></li><li><div style="margin: 6pt 0in;"><font face="Calibri">If there is already a Data Management Gateway installed on the computer you want to install the connector to, uninstall it before installing the connector. An Analysis Service Connector and a Data Management Gateway cannot exist on the same computer. </font></div></li><li><div style="margin: 6pt 0in;"><font face="Calibri">If you have multiple Analysis Services instances, you will have to install and configure a connector for each of instance. Because you cannot have more than one connector installed on a single computer, each connector will have to be installed on a different computer, and each one of those computers must remain running. This usually isn’t a problem if you install the connector on the same computer as Analysis Services with only one instance. </font></div></li><li><div style="margin: 6pt 0in;"><font face="Calibri">Remember, a single Analysis Services instance can have many tabular models. A connector is configured for a single instance, so if you have multiple tabular models, but they’re all on the same instance, you will only have to install and configure one connector. </font></div></li><li><div style="margin: 6pt 0in;"><font face="Calibri">The computer you install the connector on must remain on all the time, and the Analysis Services Connector needs to remain running. This usually isn’t a problem if you install the connector on the same computer as Analysis Services.</font></div></li><li><div style="margin: 6pt 0in;"><font face="Calibri">The Analysis Services Connector is called Data Management Gateway Host Service in Services.<span><img src="https://powerbi.uservoice.com/assets/78794056/ASConnector_DMGInServices.png"></span></font></div></li><li><div style="margin: 6pt 0in;"><font face="Calibri">.NET Framework 4.5.1 or later must be installed on the same computer as the connector. </font></div></li></ul><p style="margin: 6pt 0in;"><strong>

<br></strong>
</p><h1 style="margin: 12pt 0in 6pt;"><font color="#365f91" face="Cambria" size="5">Uploading Excel workbooks with Analysis Services connections</font></h1><p style="margin: 6pt 0in;">
<font face="Calibri">Users in Power BI can upload Excel 2013 workbooks that
already connect to Analysis Services tabular models. This can be useful when a
user has already created a workbook with Power View reports based on Analysis
Services tabular model data, and they want those reports for dashboards in
Power BI.</font></p><p style="margin: 6pt 0in;">

<span style='line-height: 107%; font-family: "Calibri",sans-serif; font-size: 11pt; mso-ansi-language: EN-US; mso-fareast-font-family: Calibri; mso-bidi-font-family: "Times New Roman"; mso-fareast-language: EN-US; mso-ascii-theme-font: minor-latin; mso-fareast-theme-font: minor-latin; mso-hansi-theme-font: minor-latin; mso-bidi-theme-font: minor-bidi; mso-bidi-language: AR-SA;'>The Analysis Services connection string
configured in the Excel workbook needs to match an Analysis Services Connector
already installed and configured for the same Analysis Services instance. Power
BI will automatically match the workbook’s connection with the connection
defined in the Analysis Services Connector.</span><br></p></font><font face="Times New Roman">

</font><h1 style="margin: 12pt 0in 6pt;"><font color="#365f91" face="Cambria" size="5">Where things can go wrong</font></h1><font face="Times New Roman">

</font><p style="margin: 6pt 0in;">Sometimes installing the connector fails. Or, maybe the connector
seems to install ok, but Power BI is still unable to connect to an Analysis
Services server. In many cases, it’s something simple, like the password for
the Server Administrator account the connector uses to sign into Analysis
Services expires. </p><font face="Times New Roman">

</font><p style="margin: 6pt 0in;">In other cases, there might be issues with type of e-mail
address users sign in with, or Analysis Services’ inability to resolve an
effective username. If you have multiple domains with trusts between them, and
your Active Directory server is in one and Analysis Services in another, this sometimes
can cause some problems. </p><font face="Times New Roman">

</font><p style="margin: 6pt 0in;">Rather than go into troubleshooting connector issues here,
we’ve put a series of troubleshooting steps into another article; <a href="http://support.powerbi.com/knowledgebase/articles/505324-troubleshooting-analysis-service-connector"><font color="#0000ff">Troubleshooting
the Power BI Analysis Service Connector</font></a>. Hopefully you won’t have any
problems. But if you do, understanding how all of this works and the
troubleshooting article should help.</p><font face="Times New Roman">

</font><font face="Times New Roman">

</font><h1 style="margin: 12pt 0in 6pt;"><font color="#365f91" face="Cambria" size="5">FAQ</font></h1><font face="Times New Roman">

</font><p style="margin: 6pt 0in;"><b style="mso-bidi-font-weight: normal;">Question:</b> Can I
use msdmpump.dll to create custom effective username mappings?</p><font face="Times New Roman">

</font><p style="margin: 6pt 0in;"><b style="mso-bidi-font-weight: normal;">Answer:</b> No. This
is not supported at this time. </p><font face="Times New Roman">

</font><p style="margin: 6pt 0in;">&nbsp;</p><font face="Times New Roman">

</font><p style="margin: 6pt 0in;"><b style="mso-bidi-font-weight: normal;">Question:</b> Can I
use the Analysis Services Connector to connect to a Multidimensional (OLAP)
instance.</p><font face="Times New Roman">

</font><p style="margin: 6pt 0in;"><b style="mso-bidi-font-weight: normal;">Answer:</b> No. This
is not supported at this time. </p><font face="Times New Roman">

</font><p style="margin: 6pt 0in;">&nbsp;</p><font face="Times New Roman">

</font><p style="margin: 6pt 0in;"><b style="mso-bidi-font-weight: normal;">Question:</b> How
often are tiles in a report in Power BI connected to Analysis Services updated?</p><font face="Times New Roman">

</font><p style="margin: 6pt 0in;"><b style="mso-bidi-font-weight: normal;">Answer:</b> About ten
minutes. Analysis Services connections are live. This doesn’t mean that a tile
will show new data every ten minutes. In-fact, most tabular models are
processed once a day, usually in the evening. This is different from
interacting with a report, where tiles and visualizations will be updated immediately as users slice and dice the data.</p><p style="margin: 6pt 0in; font-family: Calibri; font-size: medium; font-style: normal; font-variant: normal;"><span style="line-height: 1.4em;"><br></span></p></font><font face="Calibri" size="3"><font face="Times New Roman">

</font><p style="margin: 6pt 0in;"><b style="mso-bidi-font-weight: normal;">Question:</b> Can I
upload Excel workbooks with Power Pivot data models that connect to Analysis
Services tabular models? Do I need an Analysis Services Connector for this
scenario?</p><font face="Times New Roman">

</font><p style="margin: 6pt 0in;"><b style="mso-bidi-font-weight: normal;">Answer:</b> Yes, you
can upload the workbook. And, no, you don’t need an Analysis Services
Connector. But, because the data will reside in the Excel data model, reports
in Power BI based on the Excel workbook will not be live, like with a Power BI
report that connects to an Analysis Services tabular model by using an Analysis
Services Connector. In order to refresh reports in Power BI, you’d have to
re-upload an updated workbook each time. Connecting to an Excel workbook with a
Power Pivot data model residing in OneDrive is a better solution here.</p><font face="Times New Roman">

</font><p style="margin: 6pt 0in;">&nbsp;</p><font face="Times New Roman">

</font><p style="margin: 6pt 0in;"><b style="mso-bidi-font-weight: normal;">Question:</b> What if
I install the Power BI Analysis Services Connector on a computer in a different
domain from my Analysis Services server?</p><font face="Times New Roman">

</font><p style="margin: 6pt 0in;"><b style="mso-bidi-font-weight: normal;">Answer:</b> No
guarantees here. It all depends on the trust relationship between the two
domains. If the two different domains are in a trusted domain model, then the
Connector might be able to connect to the Analysis Services server and the
effective user name can be resolved. If not, either the connection or effective
username resolution can fail. </p><font face="Times New Roman">

</font><p style="margin: 6pt 0in;">&nbsp;</p><font face="Times New Roman">

</font><p style="margin: 6pt 0in;"><b style="mso-bidi-font-weight: normal;">Question:</b> If
users share dashboards connected to Analysis Services, will those
other users be able to see the data even though they might not have the same the
permissions. </p><font face="Times New Roman">

</font><p style="margin: 6pt 0in;"><b style="mso-bidi-font-weight: normal;">Answer:</b> Yes. In
Power BI, sharing means sharing your own data. This is the current
behavior, but could change at any time. </p><font face="Times New Roman">
