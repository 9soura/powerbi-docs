Create and manage relationships in Power BI Designer

	<div class="MsoNormal"><font face="Calibri"><font size="3"><span style='line-height: 115%; font-family: "Calibri",sans-serif; font-size: 11pt; mso-ansi-language: EN-US; mso-fareast-font-family: SimSun; mso-bidi-font-family: Arial; mso-fareast-language: ZH-CN; mso-ascii-theme-font: minor-latin; mso-fareast-theme-font: minor-fareast; mso-hansi-theme-font: minor-latin; mso-bidi-theme-font: minor-bidi; mso-bidi-language: AR-SA;'>When you import multiple
tables, chances are you’re going to do some analysis using data from all those
tables. Relationships between those tables are necessary in order to accurately
calculate results and display the correct information in your reports. The
Power BI Designer makes creating those relationships easy. In-fact, in most
cases you won’t have to do anything, the designer’s Autodetect feature can do
it for you. However, in some cases you might have to create relationships
yourself, or you might need to make some changes to a relationship. Either way,
it’s important to understand relationships in Power BI and how to create and
edit them.</span></font></font></div><h1 class="MsoNormal"></h1><font face="Calibri" size="3"><h1 class="MsoNormal"><font face="Times New Roman">
</font><font color="#365f91" face="Cambria" size="5">Autodetect during load</font></h1><span style='line-height: 115%; font-family: "Calibri",sans-serif; font-size: 11pt; mso-ansi-language: EN-US; mso-fareast-font-family: SimSun; mso-bidi-font-family: Arial; mso-fareast-language: ZH-CN; mso-ascii-theme-font: minor-latin; mso-fareast-theme-font: minor-fareast; mso-hansi-theme-font: minor-latin; mso-bidi-theme-font: minor-bidi; mso-bidi-language: AR-SA;'><span style='line-height: 115%; font-family: "Calibri",sans-serif; font-size: 11pt; mso-ansi-language: EN-US; mso-fareast-font-family: SimSun; mso-bidi-font-family: Arial; mso-fareast-language: ZH-CN; mso-ascii-theme-font: minor-latin; mso-fareast-theme-font: minor-fareast; mso-hansi-theme-font: minor-latin; mso-bidi-theme-font: minor-bidi; mso-bidi-language: AR-SA;'>If you query two or
more tables at the same time, when the data is loaded, the designer will attempt
to find and create relationships for you. Cardinality, Cross filter direction,
and Active properties are automatically set. The designer looks at column names
in the tables you are querying to determine if there are any potential
relationships. If there are, those relationships are created automatically. If
the designer cannot determine with a high-level of confidence there is a match,
it will not automatically create the relationship. You can still use the Manage
Relationships dialog to create or edit relationships.</span></span><h1 class="MsoNormal"><font face="Times New Roman">

tab, click <b style="mso-bidi-font-weight: normal;">Manage Relationships</b> &gt;
<b style="mso-bidi-font-weight: normal;">New</b>.</p></li><li style='color: rgb(0, 0, 0); font-family: "Calibri",sans-serif; font-size: 11pt; font-style: normal; font-weight: normal;'><p style='color: rgb(0, 0, 0); font-family: "Calibri",sans-serif; font-size: 11pt; font-style: normal; font-weight: normal; margin-top: 0in; margin-bottom: 0pt; mso-list: l0 level1 lfo1;'>In the <b style="mso-bidi-font-weight: normal;">Create
Relationship</b> dialog, in the first table drop-down list,
 select&nbsp;a table, &nbsp;and then select the column you want to use in the relationship. </p></li><li style='color: rgb(0, 0, 0); font-family: "Calibri",sans-serif; font-size: 11pt; font-style: normal; font-weight: normal;'><p style='color: rgb(0, 0, 0); font-family: "Calibri",sans-serif; font-size: 11pt; font-style: normal; font-weight: normal; margin-top: 0in; margin-bottom: 10pt; mso-list: l0 level1 lfo1;'>In the to&nbsp;second table drop-down list,&nbsp;select the other table you want in the relationship, then select the other column you

<font color="#365f91" face="Cambria" size="5"><b>Configure advanced options</b></font></h1><p style="margin: 0in 0in 10pt;"><font face="Calibri">When you create or edit a relationship, you can configure
advanced options.<span style="mso-spacerun: yes;">&nbsp; </span>By default, the designer
will automatically configure the advanced options based on a best guess. This can
be different for each relationship based on the data in the columns.</font></p><h2><span style="color: rgb(36, 63, 96); line-height: 1.4em; font-family: Cambria; font-size: medium;">Cardinality</span><br></h2><font face="Calibri"><p style="margin: 0in 0in 10pt;"><font face="Times New Roman">

</font><b style="mso-bidi-font-weight: normal;">Many to One (*:1)</b> <span style="mso-spacerun: yes;">&nbsp;</span>- This is
the most common, default type. This means the column in one table can have more
than one instance of a value, and the other related table, often know as the
Lookup table, has only one instance of a value. </p><p style="margin: 0in 0in 10pt;"><font face="Times New Roman">

</font><b style="mso-bidi-font-weight: normal;">One to One (1:1)</b> - This means the column in one table has only one
instance of a particular value, and the other related table has only one
instance of a particular value. </p><p style="margin: 0in 0in 10pt;"><font face="Times New Roman">

</font><span style='line-height: 115%; font-family: "Calibri",sans-serif; font-size: 11pt; mso-ansi-language: EN-US; mso-fareast-font-family: SimSun; mso-bidi-font-family: Arial; mso-fareast-language: ZH-CN; mso-ascii-theme-font: minor-latin; mso-fareast-theme-font: minor-fareast; mso-hansi-theme-font: minor-latin; mso-bidi-theme-font: minor-bidi; mso-bidi-language: AR-SA;'>See the Understanding advanced options section later
in this article for more details about when to change cardinality.</span></p></font><h2><font color="#243f60" face="Cambria" size="3">Cross filter direction</font></h2><p style="margin: 0in 0in 10pt;">

<font face="Calibri"><b style="mso-bidi-font-weight: normal;">Both - </b>This is the most common, default direction. This means for
filtering purposes, both tables are treated as if they are a single table.</font></p><p style="margin: 0in 0in 10pt;">

<font face="Calibri"><b style="mso-bidi-font-weight: normal;">Single – </b>This means that filtering choices in connected tables work
on the table where values are being aggregated. If you import a Power Pivot in
Excel 2013 or earlier data model, all relationships will have a single
direction.<span style="mso-spacerun: yes;">&nbsp; </span></font></p><p style="margin: 0in 0in 10pt;">

<font face="Calibri">See the Understanding advanced
options section later in this article for more details about when to change
cross filter direction. </font></p>

</font></span></font></font><h2><font face="Calibri"><font size="3"><span style="mso-spacerun: yes;"><font face="Times New Roman"><font color="#243f60" face="Cambria" size="3">Make this relationship active</font></font></span></font></font></h2><font face="Calibri"><font size="3"><span style="mso-spacerun: yes;"><font face="Times New Roman"><p style="margin: 0in 0in 10pt;">

<font face="Calibri">When checked, this means the
relationship serves as the active, default relationship. <span style="mso-spacerun: yes;">&nbsp;</span>In cases where there are more than one
relationship between two tables, the active relationship provides a way for the
designer to automatically create visuals that include both tables.</font></p><p style="margin: 0in 0in 10pt;">

<font face="Calibri">See the Understanding advanced
options section later in this article for more details about when to make a
particular relationship active. </font></p><h1><strong>
</strong><font color="#365f91" face="Cambria" size="5">Understanding relationships</font></h1><p style="margin: 0in 0in 10pt;">

<font face="Calibri">Once you have connected two tables together with a relationship,
you can work with the data in both tables as if they were a single table, freeing
you from having to worry about relationship details, or flattening those tables
into a single table before importing them into Power BI.&nbsp;&nbsp;In many
situations, Power BI can automatically create relationships for you, so manually
creating those relationships might not even be needed; however, if Power BI
can’t determine with a high-degree of certainty that a relationship between two
tables should exist, it will <i style="mso-bidi-font-style: normal;">not </i>automatically
create the relationship for you. In that case, you will need to create the
relationship.<span style="mso-spacerun: yes;">&nbsp;&nbsp;&nbsp; </span></font></p><font face="Calibri"><span style="mso-spacerun: yes;"><p style="margin: 0in 0in 10pt;">Let’s do a little tutorial, to better show you how

relationships</font></h2><p style="margin: 0in 0in 10pt;"><strong>

</strong>

</p><p style="margin: 0in 0in 10pt;"><font face="Calibri">For most relationships, the cross filter direction is set to
‘Both’.<span style="mso-spacerun: yes;">&nbsp; </span>There are, however, some more uncommon
circumstances where you might need to set this different from the default, like
if you’re importing a model from an older version of Power Pivot, where every
relationship is set to a single direction.<span style="mso-spacerun: yes;">&nbsp;
</span></font></p><p style="margin: 0in 0in 10pt;">

</p><p style="margin: 0in 0in 10pt;"><font face="Calibri">The Both setting enables the designer to treat all aspects
of connected tables as if they are a single table.<span style="mso-spacerun: yes;">&nbsp; </span>There are some situations, however, where the
designer cannot set a relationship’s cross filter direction to ‘Both’ and also
keep an unambiguous set of defaults available for reporting purposes.<span style="mso-spacerun: yes;">&nbsp; </span>If the designer doesn’t set a relationship cross
filter direction to Both, then it’s usually because it would create
ambiguity.<span style="mso-spacerun: yes;">&nbsp; </span>If the default cross filter
setting isn’t working for you, you can set it to be toward a particular table
or Both.</font></p><p style="margin: 0in 0in 10pt;">

</p><p style="margin: 0in 0in 10pt;"><font face="Calibri">Single direction cross filtering works for many
situations.<span style="mso-spacerun: yes;">&nbsp; </span>In fact, if you’ve imported
a model from Power Pivot in Excel 2013 or earlier, all of the relationships will
be set to single direction.<span style="mso-spacerun: yes;">&nbsp; </span>Single
direction means that filtering choices in connected tables work on the table
where aggregation work is happening.<span style="mso-spacerun: yes;">&nbsp;
</span>Sometimes, understanding cross filtering can be a little difficult, so
let’s look at an example.</font></p><p style="margin: 0in 0in 10pt;"><font face="Calibri"><span><img width="608" height="270" style="width: 662px; height: 280px;" src="https://powerbi.uservoice.com/assets/75826513/CrossFilterSingle.png"></span></font></p><p style="margin: 0in 0in 10pt;"><strong>

</strong>

</p><p style="margin: 0in 0in 10pt;"><font face="Calibri">With single direction cross filtering, if you create a
report that summarizes the project hours and then you can choose to summarize
(or filter) by CompanyProject, Priority or CompanyEmployee, City.<span style="mso-spacerun: yes;">&nbsp;&nbsp; </span>If however, you want to count the number of
employee per projects (a less common question), it won’t work. You’ll get a
column of values that are all the same.<span style="mso-spacerun: yes;">&nbsp; </span>In
the example below, both relationships cross filtering direction is set to a
single direction – towards the ProjectHours table:</font></p><font face="Calibri"><span><span><img src="https://powerbi.uservoice.com/assets/78054098/CandMRel_RepCrossFilterSingle.png"></span></span></font><p style="margin: 0in 0in 10pt;"><strong>



</strong></p><p style="margin: 0in 0in 10pt;"></p><font face="Calibri"><p style="margin: 0in 0in 10pt;"><font face="Times New Roman">

</font></p><p style="margin: 0in 0in 10pt;">Filter specification will flow from CompanyProject to
CompanyEmployee (as shown in the image below)<span style="mso-spacerun: yes;">&nbsp;
</span>but, it won’t flow up to CompanyEmployee.<span style="mso-spacerun: yes;">&nbsp; </span>However, if you set the cross filtering
direction to Both it will work.<span style="mso-spacerun: yes;">&nbsp; </span>The Both
setting allows the filter specification to flow up to Employee.</p><p style="margin: 0in 0in 10pt;"><font face="Times New Roman">

</font></p></font><p style="margin: 0in 0in 10pt;"></p><p style="margin: 0in 0in 10pt;">

<span><img width="631" height="281" style="width: 649px; height: 294px;" src="https://powerbi.uservoice.com/assets/75826554/CrossFilterBoth.png"></span><br></p></font><p style="margin: 0in 0in 10pt;"></p></span><p style="margin: 0in 0in 10pt;"><span style='line-height: 115%; font-family: "Calibri",sans-serif; font-size: 11pt; mso-ansi-language: EN-US; mso-fareast-font-family: SimSun; mso-bidi-font-family: Arial; mso-fareast-language: ZH-CN; mso-ascii-theme-font: minor-latin; mso-fareast-theme-font: minor-fareast; mso-hansi-theme-font: minor-latin; mso-bidi-theme-font: minor-bidi; mso-bidi-language: AR-SA;'>With the cross
filtering direction set to Both, our report now appears correct:</span></p><span style='line-height: 115%; font-family: "Calibri",sans-serif; font-size: 11pt; mso-ansi-language: EN-US; mso-fareast-font-family: SimSun; mso-bidi-font-family: Arial; mso-fareast-language: ZH-CN; mso-ascii-theme-font: minor-latin; mso-fareast-theme-font: minor-fareast; mso-hansi-theme-font: minor-latin; mso-bidi-theme-font: minor-bidi; mso-bidi-language: AR-SA;'><span></span></span><span></span><span><img src="https://powerbi.uservoice.com/assets/78054266/CandMRel_RepCrossFilterBi.png"></span><br></font><p style="margin: 0in 0in 10pt;"><span style='line-height: 115%; font-family: "Calibri",sans-serif; font-size: 11pt; mso-ansi-language: EN-US; mso-fareast-font-family: SimSun; mso-bidi-font-family: Arial; mso-fareast-language: ZH-CN; mso-ascii-theme-font: minor-latin; mso-fareast-theme-font: minor-fareast; mso-hansi-theme-font: minor-latin; mso-bidi-theme-font: minor-bidi; mso-bidi-language: AR-SA;'>Cross filtering both
directions works well for a pattern of table relationships that look like the
pattern above. This is most commonly called a star schema, like this:</span></p><span style='line-height: 115%; font-family: "Calibri",sans-serif; font-size: 11pt; mso-ansi-language: EN-US; mso-fareast-font-family: SimSun; mso-bidi-font-family: Arial; mso-fareast-language: ZH-CN; mso-ascii-theme-font: minor-latin; mso-fareast-theme-font: minor-fareast; mso-hansi-theme-font: minor-latin; mso-bidi-theme-font: minor-bidi; mso-bidi-language: AR-SA;'><span><img src="https://powerbi.uservoice.com/assets/75826625/StarSchema.png"></span></span><p style="margin: 0in 0in 10pt;"><span style='line-height: 115%; font-family: "Calibri",sans-serif; font-size: 11pt; mso-ansi-language: EN-US; mso-fareast-font-family: SimSun; mso-bidi-font-family: Arial; mso-fareast-language: ZH-CN; mso-ascii-theme-font: minor-latin; mso-fareast-theme-font: minor-fareast; mso-hansi-theme-font: minor-latin; mso-bidi-theme-font: minor-bidi; mso-bidi-language: AR-SA;'></span><span style='line-height: 115%; font-family: "Calibri",sans-serif; font-size: 11pt; mso-ansi-language: EN-US; mso-fareast-font-family: SimSun; mso-bidi-font-family: Arial; mso-fareast-language: ZH-CN; mso-ascii-theme-font: minor-latin; mso-fareast-theme-font: minor-fareast; mso-hansi-theme-font: minor-latin; mso-bidi-theme-font: minor-bidi; mso-bidi-language: AR-SA;'>Cross filtering
direction does not work well with a more general pattern often found in
databases, like in this diagram:</span></p><span style='line-height: 115%; font-family: "Calibri",sans-serif; font-size: 11pt; mso-ansi-language: EN-US; mso-fareast-font-family: SimSun; mso-bidi-font-family: Arial; mso-fareast-language: ZH-CN; mso-ascii-theme-font: minor-latin; mso-fareast-theme-font: minor-fareast; mso-hansi-theme-font: minor-latin; mso-bidi-theme-font: minor-bidi; mso-bidi-language: AR-SA;'><span><img src="https://powerbi.uservoice.com/assets/75826630/AmbiguousSchema.png"></span></span><br></span><p style="margin: 0in 0in 10pt;">

</p><p style="margin: 0in 0in 10pt;"><font face="Calibri">If you have a table pattern like this, with loops, then
cross filtering can create an ambiguous set of relationships. For instance, if
you sum up a field from Table X and then choose to filter by a field on Table Y,
then it’s not clear how the filter should travel, through the top table or the
bottom table.<span style="color: rgb(148, 54, 52); mso-themecolor: accent2; mso-themeshade: 191;"> </span><span style="mso-spacerun: yes;">&nbsp;</span>Just like with
active/inactive relationships, the designer won’t allow a relationship to be
set as Both if it will create ambiguity in reports. There are several different
ways you can deal with this, here are the two most common:</font></p><ul><li style='color: rgb(0, 0, 0); font-family: "Calibri",sans-serif; font-size: 11pt; font-style: normal; font-weight: normal;'><p style='color: rgb(0, 0, 0); font-family: "Calibri",sans-serif; font-size: 11pt; font-style: normal; font-weight: normal; margin-top: 0in; margin-bottom: 0pt; mso-list: l0 level1 lfo1;'>Delete or mark relationships as inactive to reduce ambiguity. Then you might be able to set a relationship cross filtering as Both.</p></li><li style='color: rgb(0, 0, 0); font-family: "Calibri",sans-serif; font-size: 11pt; font-style: normal; font-weight: normal;'><p style='color: rgb(0, 0, 0); font-family: "Calibri",sans-serif; font-size: 11pt; font-style: normal; font-weight: normal; margin-top: 0in; margin-bottom: 10pt; mso-list: l0 level1 lfo1;'>Bring in a table twice (with a different name the second time) to eliminate loops.<span style="mso-spacerun: yes;">&nbsp; </span>This makes the pattern of relationships like a star schema.<span style="mso-spacerun: yes;">&nbsp; </span>With a star schema all of the relationships can be set to Both.</p></li></ul><p style="margin: 0in 0in 10pt;"><strong>



</strong>

</p><font color="#365f91" face="Cambria" size="4">Wrong active relationship</font><p style="margin: 0in 0in 10pt;"><strong>

</strong>

</p><p style="margin: 0in 0in 10pt;"><font face="Calibri">When the designer automatically creates relationships, it
sometimes encounters more than one relationship between two tables.<span style="mso-spacerun: yes;">&nbsp; </span>When this happens only one of the
relationships is set to be active.<span style="mso-spacerun: yes;">&nbsp; </span>The
active relationship serves as the default relationship so that when you choose
fields from two different tables, the designer can automatically create a
visualization for you.<span style="mso-spacerun: yes;">&nbsp; </span>However, in some
cases the automatically selected relationship can be wrong.<span style="mso-spacerun: yes;">&nbsp; </span>You can use the Manage Relationships dialog
to set a relationship as active or inactive, or you can set the active
relationship in the Edit relationship dialog.<span style="mso-spacerun: yes;">&nbsp;
</span></font></p><p style="margin: 0in 0in 10pt;">

</p><p style="margin: 0in 0in 10pt;"><font face="Calibri">To ensure there’s a default relationship, the designer only
